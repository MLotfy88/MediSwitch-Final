## 🚀 الحالة الحالية (Current Status)

### ✅ تم إنجازه مؤخراً
1.  **تقسيم قاعدة البيانات (D1 Split Strategy)**:
    -   إنشاء قاعدة بيانات مستقلة `mediswitch-interactions` مخصصة للتفاعلات الضخمة (320,000+ سجل).
    -   تحرير مساحة قاعدة البيانات الرئيسية `mediswitsh-db` بحذف جداول التفاعلات منها، مما حل مشكلة "Code 7500: Exceeded maximum DB size".
    -   تحديث الـ Worker (v3.5) ليتعرف على القاعدتين ويوجه الاستعلامات آلياً.

2.  **تحصيل حجم التطبيق (App Size Optimization)**:
    -   حذف أكثر من 200 ميجابايت من ملفات الـ JSON (تفاعلات الأدوية، الأمراض، الغذاء) من مجلد `assets`.
    -   أصبح حجم التطبيق الآن حوالي **40 ميجابايت** بدل 250 ميجابايت.

3.  **إصلاح مزامنة D1 وتحسين واجهة الجرعات (Jan 2026)**:
    -   إصلاح أخطاء `UNIQUE constraint` و `SQLITE_TOOBIG` في Cloudflare D1.
    -   تحويل واجهة الجرعات (Dosage Tab) لتكون أكثر تفصيلاً بكروت مستقلة لـ (Route, Category, Dose, Frequency).
    -   إزالة التكرار في عرض التعليمات الطبية.

3.  **المعمارية الهجينة (Hybrid Data Flow)**:
    -   إضافة `InteractionRemoteDataSource` لجلب البيانات من الـ API عند الحاجة.
    -   تحديث `InteractionRepositoryImpl` ليعمل بنظام "Cache-First": البحث محلياً أولاً، ثم الجلب من الـ API وحفظ النتيجة في SQLite.
    -   إضافة وظائف `saveToCache` في `SqliteLocalDataSource` لضمان عمل التطبيق أوفلاين للتفاعلات التي تم جلبها مسبقاً.

## 💡 القرارات النشطة (Active Decisions)
-   **نموذج الهجين الذكي**: التفاعلات تُجلب عند الطلب فقط لتقليل استهلاك البيانات والمساحة، مع الاحتفاظ ببيانات الأدوية والجرعات محلياً لسرعة البحث.
-   **فصل التفاعلات**: أي بيانات ضخمة مستقبلية (مثل تفاعلات الأعشاب أو جينات الأدوية) ستوضع في قاعدة بيانات `INTERACTIONS_DB`.

## 🚧 التحديات والمخاطر المتبقية
-   **الاعتماد على الإنترنت**: لأول مرة، يحتاج المستخدم لإنترنت (لمرة واحدة) لعرض تفاعلات معينة لم يسبق جلبها. يجب توفير UI Indicator يوضح عملية التحميل.
-   **مزامنة التفاعلات**: أي تحديثات جديدة للتفاعلات في لوحة الإدارة يجب أن تتوجه للقاعدة الجديدة `mediswitch-interactions`.
